# Competition Rules — Key Points

## Must-Haves

- [x] Team of 2-4 people
- [ ] Must use Interswitch's APIs
- [ ] GitHub repository (public or accessible to judges)
- [ ] Working MVP (hosted live)
- [ ] All code built during hackathon (March 23-26)
- [ ] Judges can test and see the solution

## What Judges Check

1. GitHub commit history (proves work was done during hackathon)
2. Working deployed link
3. Interswitch API integration
4. Real problem being solved
5. Quality of solution
6. Product thinking and design

## Disqualification Risks

- Pre-existing code/projects
- Solo submission (need 2-4 people)
- Using privately compiled datasets
- Building before March 23rd
- Not using Interswitch APIs

## Allowed Before Hackathon

- ✅ Planning and ideation
- ✅ Research
- ✅ Wireframes and mockups
- ✅ Architecture documents
- ✅ Setting up accounts (GitHub, Supabase, Interswitch, Vercel)
- ✅ Reading API documentation
- ✅ Creating planning files and context docs

## NOT Allowed Before Hackathon

- ❌ Writing code
- ❌ Backend setup
- ❌ Feature implementation
- ❌ API integrations
- ❌ Database creation

## Important Dates

- Building: March 23-26, 2026
- Judges Review: March 30 - April 2, 2026
- Demo Day: April 11, 2026 (Top 10, physical at Interswitch HQ)

## Submission Requirements

1. GitHub repository link
2. Working MVP link (hosted)
3. Test credentials if needed
